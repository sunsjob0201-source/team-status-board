# Team Status Board

Team Status Board（チーム状況ボード）は、GitHubを初めて使う3〜8人のグループ向け共同作業教材です。メンバーは自分専用のJSONファイルを1つ追加し、Issueから公開確認までを30〜45分で体験します。

アプリ本体を書き換える必要はありません。ブラウザ版GitHubだけで基本コースを完了できます。

## この教材で学べること

- Issue（作業内容を記録する場所）を読み、自分を担当者にする
- ブランチ（安全に変更を分ける作業場所）を作る
- ファイルを追加してコミット（変更の記録）する
- Pull Request（変更を取り込んでもらう提案）を作る
- 差分をレビュー（確認）し、コメントへ対応する
- 自動チェックの結果を読み、承認後にマージ（変更を統合）する
- GitHub Pagesで公開結果を確認する

完成画面には、お知らせ、登録人数、完了人数、メンバーごとの「名前・担当・状態・ひとこと」が表示されます。カードは画面幅に合わせて1〜3列になります。

## 最初に見るファイル

| ファイル | 内容 |
| --- | --- |
| `members/example.json` | 参加者がコピーする見本 |
| `announcement.txt` | ページ上部のお知らせ。コンフリクト練習にも使用 |
| `CONTRIBUTING.md` | ブランチ名やレビューの約束 |
| `FACILITATOR.md` | 講師向けの準備・進行手順 |
| `scripts/validate.mjs` | JSONを初心者向けメッセージで検証 |
| `scripts/build.mjs` | 公開用の `dist/` を生成 |

## 参加者向け：30〜45分コース

講師がGitHub PagesのURLと、あなた用のIssue番号を案内してから始めます。

### 1. Issueを読み、自分を担当者にする

1. リポジトリ上部の「Issues」を開きます。
2. 講師から案内された「メンバーカードを追加する」Issueを開きます。
3. 本文の表示名・担当・状態・ひとこと・ファイル名と、完了条件を読みます。
4. 右側の「Assignees」で歯車または「assign yourself」を押し、自分を選びます。

「Assignee」は、そのIssueを担当する人のことです。

### 2. 作業ブランチを作る

1. Issueの右側にある「Development」欄の「Create a branch」を押します。
2. ブランチ名を `feature/issue-番号-短い名前` にします。Issue 12、田中さんなら `feature/issue-12-tanaka` です。
3. 作成先としてこのリポジトリを選び、「Create branch」を押します。
4. 「Code」タブへ戻り、左上のブランチ選択欄が今作ったブランチ名になっていることを確認します。

「Create a branch」が表示されない場合は、「Code」タブのブランチ選択欄から「View all branches」→「New branch」を使います。元になるブランチは `main` にします。

### 3. 自分のJSONファイルを追加する

1. 「Code」タブで `members` フォルダーを開きます。
2. `example.json` を開き、内容を確認します。
3. `members` フォルダーへ戻り、「Add file」→「Create new file」を押します。
4. ファイル名にIssueで決めた名前を入力します。例：`tanaka.json`
5. 次の4項目を入力します。各値はダブルクォート `"` で囲み、行末のカンマを忘れないでください。

```json
{
  "name": "田中",
  "role": "デザイン",
  "status": "作業中",
  "message": "トップ画面を確認しています"
}
```

`status` は `未着手`、`作業中`、`確認待ち`、`完了` のどれかをそのまま使います。ファイル名には半角小文字・数字・ハイフンだけを使えます。

### 4. 変更をコミットする

1. 入力欄右上の「Preview」または画面上の差分で、内容を見直します。
2. 「Commit changes...」を押します。
3. コミットメッセージを `feat: 田中のメンバーカードを追加` のように入力します。
4. 作業ブランチへコミットする設定を確認し、「Commit changes」を押します。

「Commit」は、誰が何を変えたか分かる形で変更を保存する操作です。

### 5. Pull Requestを作る

1. コミット後に表示される「Compare & pull request」を押します。見つからない場合は「Pull requests」→「New pull request」を押します。
2. `base` が `main`、`compare` が自分の作業ブランチであることを確認します。
3. 「Create pull request」を押します。
4. テンプレートの「変更内容」を記入します。
5. 「関連Issue」の `Closes #` の後ろにIssue番号を書きます。例：`Closes #12`
6. 自分で確認した項目だけ、`[ ]` を `[x]` にします。
7. 「Create pull request」を押します。

`Closes #12` と書くと、マージ時にIssue 12も自動で完了になります。

### 6. 自動チェックを確認する

Pull Request下部のChecks欄を見ます。「JSON検証とビルド」が緑色のチェックになれば成功です。

赤い×になった場合は「Details」を押します。ログ末尾に、対象ファイルと日本語の修正理由が表示されます。「よくあるエラー」を参考に修正してください。

### 7. 別の参加者をレビューする

講師から案内された別の参加者のPull Requestを開きます。

1. 「Files changed」を押し、緑色が追加、赤色が削除であることを確認します。
2. コメントしたい行の左に表示される「+」を押します。
3. 良い点を1つ書き、軽微な修正を1つ具体的に依頼します。
4. 「Start a review」を押します。
5. 右上の「Review changes」を押し、「Request changes」または「Comment」を選びます。
6. まとめを書いて「Submit review」を押します。

例：「`message` が作業内容を具体的に伝えていて分かりやすいです。文末に句点を追加すると、ほかのカードと表記がそろいそうです。」

### 8. コメントに対応し、再レビューを依頼する

自分のPull Requestへ戻ります。

1. レビューコメントの内容を確認します。
2. 「Files changed」から対象ファイルの鉛筆アイコン「Edit file」を押します。
3. 内容を直し、「Commit changes...」から同じ作業ブランチへ追加コミットします。
4. コメント欄へ「修正しました。再確認をお願いします」と返信します。
5. 必要なら右側の「Reviewers」で同じ人へ再レビューを依頼します。

新しいコミットは同じPull Requestへ自動で追加されます。Pull Requestを作り直す必要はありません。

### 9. 承認し、マージする

レビュアーは修正と自動チェックを確認し、「Review changes」→「Approve」→「Submit review」で承認します。

承認後、作成者または講師が「Merge pull request」→「Confirm merge」を押します。講師の指示がある場合は、参加者だけでマージせず待ってください。

### 10. 公開を確認する

1. リポジトリ上部の「Actions」を開きます。
2. 「GitHub Pagesへ公開」が緑色のチェックになるまで待ちます。
3. 講師から案内されたGitHub PagesのURLを再読み込みします。
4. 自分のカード、担当、状態、ひとことが表示されることを確認します。
5. 元のIssueが「Closed」になったことも確認します。

## JSONのルール

- `name`、`role`、`status`、`message` の4項目が必要です。
- 4項目は文字列で、空欄にできません。
- `name` と `role` は30文字以内、`message` は100文字以内です。
- 同じ `name` を複数のファイルで使えません。
- ファイル名は `tanaka-san.json` のように、半角小文字・数字・ハイフンだけで作ります。
- JSON内にパスワード、メールアドレスなどの秘密情報を書かないでください。

## よくあるエラーと対処方法

| メッセージの要点 | 確認すること |
| --- | --- |
| JSONとして読み込めません | ダブルクォート、カンマ、波かっこ `{ }` の不足を確認する |
| 必須項目がありません | 見本と同じ4項目名を追加する。大文字・小文字も同じにする |
| 文字列にしてください | 値をダブルクォートで囲む |
| statusの状態が正しくありません | 定義済みの4種類から選び、余分な空白や誤字を直す |
| 名前が重複しています | 既存カードと区別できる表示名にする |
| ファイル名が正しくありません | 日本語・大文字・空白・アンダースコアを使わず、`.json` で終える |
| Pagesに反映されません | Actionsの公開処理が完了したか、Pull Requestがマージ済みか確認する |

直し方が分からないときは、エラーログの対象ファイル名と最初の修正理由を講師へ伝えてください。

## 講師・管理者向け：検証とビルド

Node.js 24 LTS以降を使います。外部パッケージのインストールは不要です。

```text
node scripts/validate.mjs
node scripts/build.mjs
```

ビルドが成功すると、公開可能なファイルが `dist/` に生成されます。ローカル表示を確認する場合は、ビルド後に次を実行して `http://127.0.0.1:4173` を開きます。

```text
node scripts/serve.mjs
```

`index.html` を直接ダブルクリックすると、ブラウザの制限で `data.json` を取得できない場合があります。上記のローカルサーバーを使ってください。

## 発展編：ローカルでclone、pull、pushする

この項目はGitとNode.jsを自分のPCへ準備した参加者向けです。基本コースでは不要です。

1. リポジトリの「Code」を押し、HTTPSのURLをコピーします。
2. ターミナルで `git clone コピーしたURL` を実行します。
3. `cd リポジトリ名` でフォルダーへ移動します。
4. `git switch main`、`git pull origin main` で最新の `main` を取得します。
5. `git switch -c feature/issue-番号-短い名前` で作業ブランチを作ります。
6. ファイルを変更し、`node scripts/validate.mjs` で確認します。
7. `git status` と `git diff` で変更範囲を確認します。
8. `git add members/自分のファイル.json` で変更を選びます。
9. `git commit -m "feat: 自分のメンバーカードを追加"` で記録します。
10. `git push -u origin 作業ブランチ名` でGitHubへ送ります。
11. GitHubに表示される「Compare & pull request」からPull Requestを作ります。

他の人の変更を取り込むときは、作業中のファイルを保存・コミットしてから `git pull` を実行します。分からない状態で強制pushは行わないでください。

## 自動化の仕組み

- Pull Request作成・更新時：`scripts/validate.mjs` と `scripts/build.mjs` を実行します。
- `main` へのマージ時：`dist/` を作り、GitHub Pagesへ公開します。
- `dist/` は生成物なのでGitへコミットしません。
- ブラウザは公開済みの `data.json` を読み込み、文字列として安全にカードを表示します。

利用開始前のGitHub設定、Issueの準備、ブランチ保護、時間配分は [FACILITATOR.md](./FACILITATOR.md) を参照してください。
