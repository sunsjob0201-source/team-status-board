# 講師向け進行ガイド

この文書だけで、3〜8人向けのTeam Status Board演習を準備・進行できるようにまとめています。GitHubのプランやOrganizationのポリシーにより、設定名や利用できるルールが異なる場合があります。

## 開催前チェックリスト

- [ ] このリポジトリ一式を演習用のGitHubリポジトリへ配置した
- [ ] 既定ブランチ名が `main` になっている
- [ ] Actionsの実行が許可されている
- [ ] GitHub Pagesの公開元を「GitHub Actions」に設定した
- [ ] `node scripts/validate.mjs` と `node scripts/build.mjs` が成功した
- [ ] `main` へのpush後、「GitHub Pagesへ公開」が成功した
- [ ] 公開URLでお知らせ・集計・2枚の初期カードを確認した
- [ ] 参加者をCollaboratorとして招待した
- [ ] 参加者ごとのIssueを作成した
- [ ] レビュー相手とマージ担当を決めた
- [ ] `main` のPull Request・承認・チェックに関するルールを設定した
- [ ] トラブル時に使う予備Issueと、復旧担当者を決めた

## 1. GitHub Pagesを有効にする

1. リポジトリの「Settings」を開きます。
2. 左側の「Pages」を開きます。
3. 「Build and deployment」の「Source」で「GitHub Actions」を選びます。
4. 「Actions」タブを開き、「GitHub Pagesへ公開」を「Run workflow」から手動実行するか、`main` へ初回pushします。
5. ワークフロー完了後、「Settings」→「Pages」に表示される公開URLを開きます。

公開URLは参加者へ事前共有します。初回公開には少し時間がかかることがあります。

## 2. 参加者を招待する

1. 「Settings」→「Collaborators and teams」または「Collaborators」を開きます。
2. 「Add people」を押します。
3. 参加者のGitHubユーザー名またはメールアドレスを検索します。
4. 演習に必要な書き込み権限を付けて招待します。
5. 参加者に招待メールまたはGitHub通知から承認してもらいます。

Organizationでは、講師に招待権限がない場合があります。その場合は管理者へ事前に依頼してください。公開リポジトリでも、ブランチ作成やPull Requestのために適切な権限が必要です。

## 3. `main` を保護する

リポジトリ内のファイルだけでは保護ルールを有効化できません。GitHub画面で設定します。

推奨するRulesetでの手順：

1. 「Settings」→「Rules」→「Rulesets」を開きます。
2. 「New ruleset」→ブランチ用のRulesetを選びます。
3. 対象ブランチに既定ブランチ `main` を含めます。
4. Rulesetを「Active」にします。
5. 「Require a pull request before merging」を有効にします。
6. 必要な承認数を1件にします。
7. 「Require status checks to pass」を有効にし、`JSON検証とビルド` を必須チェックにします。
8. 可能なら、例外対象を講師だけに限定し、参加者による直接pushを禁止します。
9. 保存後、テスト用Pull Requestで期待どおりブロックされることを確認します。

必須チェック名は、検証ワークフローが一度実行されるまで候補に出ない場合があります。その場合はテスト用Pull Requestを先に作ります。

Rulesetが使えない場合は、「Settings」→「Branches」→「Add branch protection rule」で `main` を指定し、次を有効にします。

- Require a pull request before merging
- Require approvals：1件
- Require status checks to pass before merging
- 対象チェック：`JSON検証とビルド`
- 利用可能なら、管理者を含むルール適用や直接push制限

個人プラン、Organization設定、private/publicの違いにより、一部の保護機能が利用できないことがあります。使えない場合は、講師がマージ担当となり「`main` へ直接コミットしない」運用で補います。

## 4. 参加者用Issueを用意する

1. 「Issues」→「New issue」を押します。
2. 「メンバーカードを追加する」を選びます。
3. 参加者の表示名、担当、開始時の状態、ひとこと、ファイル名を入力します。
4. タイトルに参加者名を追加します。例：`[メンバー追加] 田中さん`
5. Issueを作成し、番号を進行表へ記録します。
6. 人数分繰り返します。担当者設定は、参加者本人に演習中に行ってもらいます。

GitHubのIssue複製機能が利用できる場合は最初のIssueを複製して内容を変更しても構いません。利用できない場合はIssueフォームから人数分作ります。

表示名は重複させず、ファイル名は `tanaka.json` のように半角小文字で用意します。軽微なレビュー依頼を作りやすくするため、最初のひとことは参加者が自由に変えられる余地を残します。

## 5. 3〜8人向け時間配分

| 時間 | 内容 | 講師の確認 |
| --- | --- | --- |
| 0〜5分 | 目的、完成画面、用語を説明 | 全員が公開URLと自分のIssueを開けたか |
| 5〜10分 | Issue担当、ブランチ作成 | `main` ではなく作業ブランチか |
| 10〜18分 | JSON追加、コミット | ファイル名と4項目が正しいか |
| 18〜23分 | Pull Request作成、自動チェック | `Closes #番号` とチェック結果 |
| 23〜30分 | 相互レビュー、修正依頼 | 全員がレビューをする側も体験したか |
| 30〜36分 | 修正、再レビュー、承認 | 追加コミットと会話が残ったか |
| 36〜41分 | 順番にマージ、Pages確認 | 公開ワークフローの完了を待つ |
| 41〜45分 | 振り返り | 学びと疑問を1人ずつ共有 |

5人以上では、JSON入力中の質問対応が重なります。講師以外にサポート役を1人置くか、早く終わった参加者にログの読み方を案内してもらいます。

## 6. レビュー相手の組み合わせ

参加者を輪にして「自分の次の人」をレビューすると、漏れを防げます。

- 3人：A→B、B→C、C→A
- 4人：A→B、B→C、C→D、D→A
- 5〜8人：同じ要領で最後の人→A

レビューでは、良い点を1つ、軽微で具体的な修正依頼を1つ書いてもらいます。データの意味を大きく変える依頼や、相手を責める表現は避けます。

## 7. コンフリクト課題（追加10〜15分）

基本コース完了後に実施します。

1. `announcement.txt` の現在の1行を参加者全員で確認します。
2. 全員が同じ `main` から別々のブランチを作ります。例：`practice/conflict-a`
3. 全員が1行目を異なるお知らせに変更し、Pull Requestを作ります。
4. 最初の1人のPull Requestだけをマージします。
5. 残りのPull Requestでコンフリクト表示を確認します。
6. 「Resolve conflicts」が使える場合は押し、両方の案を相談して1行へまとめます。
7. `<<<<<<<`、`=======`、`>>>>>>>` の印をすべて消し、「Mark as resolved」→「Commit merge」を押します。
8. 自動チェックの成功後に再レビューし、順番にマージします。

全員が同時に解消すると再度競合しやすいため、2人目以降は1件ずつ進めます。時間が足りない場合はコンフリクト画面の確認までで終了して構いません。

## 8. トラブル時の復旧

### ブランチを作らず `main` を編集した

コミット前なら画面を戻り、先に作業ブランチを作ります。コミット済みで保護されていなかった場合は、以降の作業を止めて講師が変更内容を確認します。履歴を書き換える強制操作はせず、必要なら元に戻すPull Requestを作ります。

### 自動チェックが失敗した

Pull Requestの「Details」からログ末尾を開き、最初の日本語エラーを1件ずつ直します。対象ファイル名、項目名、修正理由を参加者本人に読んでもらいます。

### 作業ブランチを間違えた

まだコミットしていなければ正しいブランチへ切り替えます。コミット済みなら削除せず、講師へ知らせます。変更が小さければ正しいブランチでファイルを作り直し、不要なPull Requestを理由付きで閉じます。

### Pull Requestに別の変更が混ざった

「Files changed」で範囲を確認します。初心者演習では複雑な履歴修正を避け、正しい `main` から新しいブランチを作って必要な1ファイルだけ追加し直す方法を優先します。

### Pages公開が失敗した

1. 「Actions」→失敗した「GitHub Pagesへ公開」→失敗ステップを確認します。
2. Buildの失敗ならJSONエラーを直すPull Requestを作ります。
3. Deployの失敗なら「Settings」→「Pages」のSourceが「GitHub Actions」か確認します。
4. Actionsの権限やOrganizationポリシーでPagesが禁止されていないか管理者へ確認します。
5. 一時的な失敗ならワークフローの「Re-run jobs」を1回試します。

### 参加者が操作できない

招待を承認済みか、正しいGitHubアカウントでログインしているか、リポジトリへの権限があるかを確認します。権限設定を下げる・上げる操作は、Organization管理者の方針に従います。

## 9. 終了後の振り返り質問

- Issueがあることで、作業開始前に何が明確になりましたか。
- ブランチを分ける理由を、自分の言葉でどう説明しますか。
- Pull Requestの差分で、どこを確認しましたか。
- 受け取りやすかったレビューコメントはどんなものでしたか。
- 自動チェックは、人のレビューと何が違いましたか。
- マージから公開まで、どの処理が自動化されていましたか。
- コンフリクトが起きたとき、コード以外にどんな相談が必要でしたか。
- 実際のチームでこの流れを使うなら、どんなルールを追加したいですか。

## 10. 終了後

- 未完了のIssueとPull Requestを確認します。
- 演習用リポジトリを残す期間を参加者へ伝えます。
- Collaborator権限を継続する必要がなければ、組織の方針に従って解除します。
- 個人情報や秘密情報が誤ってコミットされていないか確認します。見つかった場合は、単にファイルを削除するだけでなく、組織のセキュリティ担当へ相談します。
- 次回用に、詰まりやすかった手順と所要時間を記録します。
